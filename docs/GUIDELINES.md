<h1 align="center">Brand Guidelines</h1>

**The Lupaxa Project** branding represents the identity of the entire ecosystem. Consistent use of logos, colours and supporting assets helps create a
professional and recognisable experience across repositories, documentation, websites, presentations and other public-facing materials.

These guidelines describe the recommended practices for using the branding assets contained within the **Brand Assets** repository.

## Guiding Principles

All branding should strive to be:

- Consistent
- Simple
- Professional
- Recognisable
- Accessible
- High quality

Every repository should present a common visual identity regardless of which organisation it belongs to.

## Official Assets

Only assets contained within the **Brand Assets** repository should be considered official.

Do not:

- recreate logos
- recolour logos
- redraw artwork
- recreate typography
- create unofficial variations

When updates are required they should be made centrally and then referenced by downstream repositories.

## Logo Variants

The standard organisation asset set consists of three images.

| Asset                  | Purpose                   |
| :--------------------- | :------------------------ |
| `readme-logo.png`      | Repository README files.  |
| `logo-transparent.png` | General-purpose branding. |
| `logo-white.png`       | Dark backgrounds.         |

Some organisations may include additional variants where required.

## Choosing the Correct Logo

| Situation       | Recommended Asset         |
| :-------------- | :------------------------ |
| GitHub README   | `readme-logo.png`         |
| Documentation   | `logo-transparent.png`    |
| Website         | `logo-transparent.png`    |
| Presentation    | `logo-transparent.png`    |
| Dark background | `logo-white.png`          |
| Social media    | Most appropriate variant. |

## Backgrounds

Always select a logo appropriate for the background.

### Light backgrounds

Use:

- `logo-transparent.png`

Avoid:

- white logos
- low-contrast artwork

### Dark backgrounds

Use:

- `logo-white.png`

Avoid:

- transparent logos that become difficult to distinguish.

## Logo Scaling

Logos should always retain their original proportions.

### Correct

- Scale proportionally.
- Preserve aspect ratio.
- Export at an appropriate resolution.

### Incorrect

- Stretch horizontally.
- Stretch vertically.
- Compress unevenly.
- Rotate unnecessarily.

## Minimum Size

Logos should remain easily recognisable.

Recommended minimum sizes.

| Asset         | Minimum Width |
| :------------ | ------------: |
| README logo   | 128 px        |
| Documentation | 128 px        |
| Website       | 256 px        |
| Presentation  | 256 px        |
| Social media  | 512 px        |

Smaller sizes may reduce readability.

## Clear Space

Maintain adequate whitespace around every logo.

A good rule is to leave at least the height of the "L" in the Lupaxa wordmark as clear space on every side.

Logos should never appear crowded by:

- text
- page borders
- icons
- buttons
- other logos

## Cropping

Do not crop logos.

Every logo should remain fully visible.

Avoid:

- clipping edges
- partial logos
- decorative cropping
- circular crops unless specifically designed for that purpose

## Rotation

Logos should always remain upright.

Do not:

- rotate
- skew
- distort
- mirror

## Colours

Use only the official Lupaxa colour palette.

Do not:

- recolour logos
- apply gradients
- add shadows
- add outlines
- add glow effects

Colour definitions are documented in **COLOURS.md**.

## Transparency

Prefer transparent logo variants wherever possible.

Transparent assets work well for:

- README files
- websites
- presentations
- documentation
- diagrams

## Typography

Where accompanying text is required:

- use clean modern typography
- maintain generous spacing
- avoid decorative fonts
- prioritise readability

Typography should complement the branding rather than compete with it.

## README Usage

Repository README files should normally begin with the standard header.

```html
<p align="center">
    <img src="https://raw.githubusercontent.com/the-lupaxa-project/brand-assets/master/logos/organisations/the-lupaxa-project/readme-logo.png" alt="The Lupaxa Project Logo"/>
</p>

<h1 align="center">
    Repository Name
</h1>
```

This creates a consistent appearance across repositories.

## Documentation Sites

Documentation should normally use:

- transparent logo
- Dark Blue navigation
- White content background
- Light Blue hyperlinks

Maintain consistency across all documentation sites.

## Presentations

For presentations:

- use the 512 px assets where practical
- preserve whitespace
- avoid overly busy slide backgrounds
- prefer white or dark blue backgrounds

## GitHub Organisation Branding

Every GitHub organisation should provide the standard organisation asset set.

New organisations should follow the same naming convention.

```txt
readme-logo.png
logo-transparent.png
logo-white.png
```

This ensures consistency across the ecosystem.

## Accessibility

Branding should remain accessible to all users.

Recommendations include:

- maintain high colour contrast
- avoid relying on colour alone
- preserve adequate whitespace
- avoid low-resolution assets
- ensure logos remain recognisable at common display sizes

## File Naming Standards

Core branding assets should use descriptive filenames.

Examples include:

```text
main-logo-transparent.png
main-logo-white.png
banner-transparent.png
banner-white.png
```

Organisation branding should use:

```text
readme-logo.png
logo-transparent.png
logo-white.png
```

Consistent naming makes assets predictable and simplifies automation.

## Summary

Good branding is achieved through consistency.

When creating documentation or other public-facing material:

- use official assets
- use the correct logo variant
- preserve proportions
- preserve whitespace
- use the official colour palette
- keep layouts simple and uncluttered

Following these guidelines helps ensure every repository presents a consistent and professional identity throughout **The Lupaxa Project**.
